# Transcom Deals UX

## Overview

Use this skill when the user wants the deals screen to be easier to scan, filter, compare, or act on.

## Where The Deals UI Lives

- Main generator and inline UI script: `analytics.js`
- Deals view renderer: search for `renderDealsV2(`
- Shared report styles are embedded in the HTML generated by `analytics.js`

## UX Priorities

- Make the list easy to scan before making it flashy
- Surface the most useful signals in the card header
- Keep filters close to the list they control
- Prefer summary first, detail on expand
- Preserve readability with large datasets and long note text
- Keep Russian UI copy short and operational

## Good Patterns For This Repo

- Sticky filter bars for long deal lists
- Compact KPI blocks in card headers
- Sorting by recent activity, score, amount, or urgency
- Focus filters like `new`, `calls`, `analyses`, `transcripts`, `notes`
- Collapsible sections for transcripts and notes
- Generated summaries for last touch and latest note

## Implementation Rules

- Edit `analytics.js`, not `report.html`
- Keep the generated page self-contained
- Avoid adding external dependencies for simple UI behavior
- Keep CSS class naming consistent with the existing inline style block
- After changes, rebuild HTML and verify the report in browser

## Verification

1. `node --check analytics.js` using the bundled runtime if needed
2. Rebuild with `analytics.js --html`
3. Verify that the inline script in `report.html` parses correctly
4. Open the local server and test the deals tab interactions
