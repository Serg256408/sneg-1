# ТрансКом — Правила оценки скрипта продаж

## I. Система сбора данных

### Источники данных (Planfix CRM)
- **Задачи (сделки)** — основные сделки менеджера
- **Подзадачи** — все данные из подзадач мёржатся в родительскую сделку (звонки, комментарии, анализы)
- **Контрагенты/Контакты** — звонки из карточки контрагента подтягиваются к сделке
- **Комментарии** — от ВСЕХ сотрудников (КП может отправить другой менеджер). Исключаются только комментарии с ИИ-рекомендациями
- **DataTags** — записи звонков (тег "Звонки") и анализы (тег "Анализ звонка")

### Приоритет загрузки
1. Сначала загружаются сделки с активностью за отчётный день
2. Затем остальные активные сделки до лимита
3. Все комментарии, звонки и анализы подзадач включаются в родительскую сделку

### Дедупликация
- Звонки в пределах 5 минут считаются дубликатами
- При мёрже подзадач проверяется уникальность записей

## II. Скрипт продаж — шаблон "Вывоз снега"

### A. Устная презентация компании (verbalPresentation)

Менеджер должен рассказать клиенту о компании по 5 пунктам:

| # | Пункт | Что должен сказать/написать |
|---|-------|-----------------------------|
| 1 | **Работаем с 2014 года** (since2014) | Компания на рынке с 2014 года, большой опыт |
| 2 | **Много объектов по Москве** (manyObjects) | Обслуживаем множество объектов в Москве |
| 3 | **Госдума и госучреждения** (govClients) | Работаем с госструктурами, Госдума |
| 4 | **Надёжность в снегопады** (reliableInSnow) | Не подводим даже в сильные снегопады |
| 5 | **Много техники** (manyVehicles) | Большой парк спецтехники |

**Оценка:**
- Сказано в **звонке** (есть в транскрибации) → `source: "call"` → **3 балла**
- Написано в **комментарии/переписке** → `source: "text"` → **1.5 балла**
- Не найдено → `source: "none"`, `done: false` → **0 баллов**

### B. Как мы работаем (howWeWork)

Менеджер должен **детально** объяснить клиенту весь процесс работы на объекте — что конкретно будет происходить, какая техника приедет, что будут делать рабочие. Клиент должен понять, что его ждёт.

**Примеры правильного описания:**
- **Вывоз снега**: "Приезжает самосвал 20 м³ и трактор-экскаватор-погрузчик. Трактор чистит территорию, грузит снег в самосвал. Вывозим на официальную сухую свалку, талоны предоставляем. У вас чистая территория."
- **Укладка асфальта**: "Приезжает бригада, укладывает асфальт марки [X]. Техника: каток, асфальтоукладчик. Вывоз строительного мусора включён. Контроль качества на каждом этапе."

**НЕ считается "Как мы работаем":**
- Общие фразы: "работаем по договору", "у нас договор с Мосводоканалом" — это информация о компании, НЕ описание процесса
- Просто упоминание техники без описания процесса
- Условия оплаты, сроки договора

**Суть**: клиент должен получить **картину** — что приедет на его объект и что там будет происходить

**Оценка:**
- В звонке (транскрибация) → `source: "call"` → **3 балла**
- В переписке → `source: "text"` → **1.5 балла**
- Не найдено → **0 баллов**

### C. Коммерческое предложение — КП (cp)

- Отправлен файл PDF с коммерческим предложением
- Искать в `[Файлы:]`: "кп", "коммерческое предложение", "КП_", "К.П."
- **1 балл**

### D. Счёт (invoice)

- Отправлен файл PDF со счётом
- Искать в `[Файлы:]` и тексте: "счёт", "счет", "invoice"
- **1 балл**

### E. Презентация — файл (writtenPresentation)

- Отправлен файл PDF с презентацией компании
- Искать в `[Файлы:]`: "презентация", "карточка компании", "presentation"
- **1 балл**

### F. Призыв к действию (callToAction)

Менеджер должен в разговоре предложить конкретное действие — **словами подтолкнуть клиента к заказу**:
- "Давайте вывозить"
- "Давайте я поставлю вас в график"
- "Мы готовы работать, когда приезжать?"
- "Давайте сделаем, у вас всё будет хорошо"
- "Давайте запланируем вывоз на завтра"
- "Когда удобно приехать на замер?"

**НЕ считается призывом:**
- "Жду вашего решения" — пассивное ожидание
- "Будем рады сотрудничеству" — вежливость
- "Обращайтесь если что" / "Пишите если понадобится" — пассив

Суть: **активные конкретные слова**, подталкивающие клиента к заказу ПРЯМО СЕЙЧАС

**Оценка:** **3 балла** (только если есть доказательство в транскрибации или переписке)

### G. Отработка возражений (objectionHandling)

Если клиент говорит:
- "Дорого" → предложить альтернативные варианты, скидку, объяснить ценность
- "Мы уже сами/есть подрядчик" → предложить лучшие условия, показать преимущества
- "Не нужен вывоз в этом году" → предложить другие услуги, оставить контакт на будущее

Суть: **не сдаваться, уговаривать, предлагать альтернативы**

**Оценка:** бонусный пункт, в баллы ЗП не входит, но отображается в отчёте

## II-B. Скрипт продаж — шаблон "Сделка" (асфальтирование)

> Применяется для сделок, название которых **НЕ начинается** с "Вывоз снега".

### A. Устная презентация компании (verbalPresentation)

| # | Ключ | Что должен сказать/написать |
|---|------|-----------------------------|
| 1 | **since2014** | Работаем с 2014 года, большой опыт |
| 2 | **fiveBrigades** | 5 бригад разной квалификации, геодезист и проектировщик в штате |
| 3 | **fullCycle** | Беремся от нуля до полного цикла: стоянки, площадки, коммерческая недвижимость |
| 4 | **bigProjects** | Крупные объекты: Микояновский мясокомбинат, ЖК Рафинад 25 тыс м², Западная долина 20 тыс м² |
| 5 | **guarantee** | Гарантия на работы + личный менеджер + бригадир на объекте + фото-отчёт |

### B. Как мы работаем (howWeWork)

**Пример правильного описания для асфальта:**
"Геодезист делает замеры, проектировщик готовит проект. Бригада готовит основание, укладывает асфальт, каток утрамбовывает. Бригадир на объекте контролирует каждый этап, делаем фото-отчёт."

### C–G. Остальные пункты — идентичны шаблону "Вывоз снега"

Призыв к действию (асфальт): "давайте сделаем замер", "давайте пришлю геодезиста", "давайте обсудим проект", "когда удобно приехать на объект?", "давайте составим смету".

## IV. Правила определения источника (КРИТИЧЕСКИ ВАЖНО)

### source: "call"
- ТОЛЬКО если конкретные слова **ДОСЛОВНО** присутствуют в секции **ТРАНСКРИБАЦИИ ЗВОНКОВ**
- Пример: транскрибация содержит "мы работаем с 2014 года" → `source: "call"`
- Звонок **БЕЗ транскрибации** = содержание **НЕИЗВЕСТНО** (source НЕ может быть "call")

### source: "text"
- Информация найдена в секции **ПЕРЕПИСКА/КОММЕНТАРИИ** или **ЗАМЕТКАХ** менеджера
- Заметка менеджера "рассказала о компании" после звонка = `source: "text"` (НЕ "call"!)
- Описание действий менеджера в текстовом виде = "text"

### source: "none"
- Информация **НЕ НАЙДЕНА** нигде в данных
- Обязательно: `done: false`

### Ключевое правило: ЗАМЕТКА ≠ ЗВОНОК
Заметка менеджера, написанная после звонка (например "обновила информацию о нас"), описывает **действие**, но НЕ является доказательством того, что информация была **произнесена** в звонке. Для `source: "call"` нужна **транскрибация** с конкретными словами.

### Правило валидации
Перед установкой `source: "call"`:
1. Существует ли секция ТРАНСКРИБАЦИИ с реальным текстом?
2. Содержит ли транскрибация конкретные слова по данному пункту?
Если хотя бы один ответ "нет" → `source` НЕ может быть "call"

## V. ИИ-итог по сделке

Для каждой сделки за каждый день ИИ должен выдать:
1. **todaySummary** — что произошло за день (2-3 предложения)
2. **missing** — что НЕ выполнено из скрипта
3. **recommendations** — конкретные рекомендации менеджеру
4. **overallVerdict** — краткий вердикт по сделке (1-2 предложения)

Рекомендации должны быть **конкретными и действенными**:
- "Отправить КП клиенту"
- "Перезвонить и рассказать о компании подробнее"
- "Предложить альтернативную цену"

---

## VI. Формат данных для ИИ

Данные разделены на чёткие секции:
1. `=== ПЕРЕПИСКА/КОММЕНТАРИИ ===` — текстовые заметки менеджера
2. `=== ЗВОНКИ (факт) ===` — метаданные звонков (дата, время, длительность)
3. `=== ТРАНСКРИБАЦИИ ЗВОНКОВ ===` — что РЕАЛЬНО СКАЗАНО по телефону
4. `=== АНАЛИЗЫ PLANFIX ===` — автоматические анализы из CRM
5. `=== ДЕЙСТВИЯ ЗА [ДАТУ] ===` — хронология действий за конкретный день

**ВАЖНО**: Звонки БЕЗ транскрибации помечаются как "СОДЕРЖАНИЕ НЕИЗВЕСТНО"

## VII. Техническая архитектура

### Файлы проекта
- **`analytics.js`** — основной скрипт: сбор данных Planfix + ИИ-оценка + генерация report.html
- **`run_daily.bat`** — автозапуск с текущей датой (Windows Task Scheduler, 18:00 ежедневно)
- **`ai_cache.json`** — кэш ИИ-оценок (ключ: `assess_{dealId}_{date}_v{N}`)
- **`latest_data.json`** — последние загруженные данные из Planfix
- **`report.html`** — HTML-отчёт для просмотра
- **`daily_log.txt`** — лог автозапусков
- **`SALES_SCRIPT_RULES.md`** — этот файл (правила оценки)

### ИИ-модель
- **DeepSeek V3** (`deepseek-chat`) — основная модель для оценки сделок
- API: `https://api.deepseek.com/v1/chat/completions` (OpenAI-совместимый формат)
- Retry: 3 попытки с паузой при ошибках
- Fallback: OpenAI GPT-4o (переключение через параметр модели в `analytics.js`)

### Кэширование
- Ключ кэша: `assess_{dealId}_{date}_v{N}` (N = версия промпта)
- При изменении правил оценки → увеличить версию → все оценки пересчитаются
- **Текущая версия: v17**
- Кэш НЕ нужно чистить вручную — старые версии просто игнорируются

### Пост-валидация (программная защита от галлюцинаций)
- Если у сделки НЕТ транскрибаций → `source: "call"` автоматически заменяется на `source: "text"`
- Баллы пересчитываются после коррекции
- Это гарантия — даже если ИИ ошибётся, программа исправит

### Запуск вручную
```
node analytics.js "Боровая" 13-03-2026
```
- Первый аргумент: фамилия менеджера
- Второй аргумент: дата отчёта (DD-MM-YYYY)

### Автозапуск
- Windows Task Scheduler: задача `TransKom_DailyReport`
- Время: 18:00 ежедневно
- Скрипт: `c:\transcom\run_daily.bat`

## VIII. Частые ошибки ИИ и как они решены

| Ошибка | Решение |
|--------|---------|
| ИИ ставит `source:"call"` без транскрибации | Правило №4 в промпте + пост-валидация в коде |
| "Жду решения" = призыв к действию | Явные примеры НЕ-призывов в промпте |
| "Договор с Мосводоканалом" = "Как мы работаем" | Примеры правильного и неправильного в промпте |
| Заметка менеджера = доказательство звонка | Правило "ЗАМЕТКА ≠ ЗВОНОК" в промпте |
| Дублирование звонков | Дедупликация в пределах 5 минут в коде |
